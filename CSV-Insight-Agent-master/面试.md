# CSV-Insight-Agent 面试文档

## 1. 项目一句话介绍

CSV-Insight-Agent 是一个基于 `LangChain Agent Loop + LangGraph + Skill Registry + MemoryStore` 的 CSV 数据分析智能体。用户上传 CSV 并输入自然语言需求后，系统可以自动完成数据画像、图表规划、图表生成、中文洞察、Markdown/HTML/PDF 报告导出，并记录历史任务和图表偏好。

推荐面试表述：

> 我做了一个 CSV 数据分析智能体，不是简单调用大模型聊天，而是把数据分析拆成多个可控工具，由 LangGraph 固定工作流或 LangChain ReAct Agent Loop 自动编排执行，最终生成图表、洞察和报告。

---

| 报告 | Markdown / HTML / WeasyPrint PDF |

| 测试 | pytest |

## 3. 项目总体架构

```text
Streamlit UI / CLI
  -> create_graph_workflow()
  -> LangGraph Workflow
  -> SkillRegistry / LangChain Tools
  -> pandas / matplotlib / seaborn / LLMClient / MemoryStore
  -> outputs/charts + outputs/reports + memory/task_history.jsonl
```

分层设计：

```text
交互层：app.py / main.py
编排层：LangGraph + LangChain Agent Loop
能力层：SkillRegistry + Skills
状态层：GraphState + MemoryStore + outputs
```

核心思想：

1. UI 与分析逻辑分离；
2. LLM 不直接执行任意代码；
3. 所有可执行能力都封装为白名单 Skill；
4. 固定工作流负责稳定产物，Agent Loop 展示自主决策；
5. MemoryStore 保存历史任务和图表偏好。

## 6. 每个模式的调用链

## 6.1 Quick Chart Mode：单图 + 中文解释

适合：快速探索 CSV，生成一张最合适图表和中文解释。

调用链：

```text
load_memory_context
  -> profile_csv
  -> route_task
  -> plan_single_chart
  -> execute_chart
  -> explain_chart
  -> finalize
  -> END
```

步骤：

1. `plan_single_chart` 调用 `SuggestChartSkill`，基于 profile 和 query 推荐图表，取第一张；
2. `execute_chart` 调用 `PlotChartSkill`，生成 PNG 图表；
3. `explain_chart` 调用 `GenerateInsightSkill`，生成中文解释；
4. `finalize` 保存任务历史和图表偏好。

产物：

```text
1 张 PNG 图表
中文解释 final_answer
memory/task_history.jsonl 任务记录
```

面试讲法：

> Quick Chart 是最轻量、稳定的路径，适合用户快速看一个 CSV 的核心趋势。

---

## 6.2 Full Report Mode：多图 + 完整报告 + PDF/HTML

适合：生成完整分析报告。

调用链：

```text
load_memory_context
  -> profile_csv
  -> route_task
  -> plan_multi_charts
  -> execute_chart_batch
  -> generate_insights
  -> draft_report
  -> safety_check
  -> export_report
  -> finalize
  -> END
```

步骤：

1. `plan_multi_charts` 调用 `SuggestChartSkill` 推荐多张图，最多取 6 张；
2. `execute_chart_batch` 调用 `PlotChartBatchSkill` 批量生成图表；
3. `generate_insights` 调用 `GenerateInsightSkill` 生成中文洞察；
4. `draft_report` 调用 `DraftMarkdownReportSkill` 生成 Markdown 报告；
5. `safety_check` 检查报告是否存在明显幻觉或字段错误；
6. `export_report` 调用 `ExportPDFSkill` 导出 Markdown、HTML、PDF；
7. `finalize` 保存任务历史。

产物：

```text
多张 PNG 图表
analysis_insights
Markdown 报告
HTML 报告
PDF 报告或 HTML fallback
```

面试讲法：

> Full Report 是最稳定的完整产物路径。它使用 LangGraph 固定流程，适合真实用户使用和项目演示。

---

## 6.3 Planner Loop Mode：JSON 工具规划

适合：展示 LLM 如何一步步规划工具调用。

调用链：

```text
load_memory_context
  -> profile_csv
  -> route_task
  -> json_planner_loop
      -> PlannerLoopRunner
          -> LLM 输出 JSON action
          -> 校验 action
          -> SkillRegistry.call(tool_name, **args)
          -> 合并结果到 state
          -> 把 observation 追加回 LLM
          -> 循环直到 final_answer / max_steps / fallback
  -> finalize
  -> END
```

LLM 必须输出：

```json
{
  "thought": "下一步思考",
  "tool_name": "suggest_chart",
  "tool_args": {}
}
```

结束时输出：

```json
{
  "thought": "任务完成",
  "tool_name": "final_answer",
  "tool_args": {"answer": "中文总结"}
}
```

可调用工具：

```text
profile_csv
suggest_chart
plot_chart
plot_chart_batch
generate_insight
draft_markdown_report
export_pdf
read_recent_memory
save_memory
final_answer
```

稳定性设计：

```text
max_steps = 8
JSON 解析失败 -> repair prompt
连续失败 2 次 -> deterministic fallback
已有图表但缺洞察 -> generate_insight
已有洞察但缺报告 -> draft_markdown_report
已有报告但缺导出 -> export_pdf
```

面试讲法：

> Planner Loop 是我自己实现的轻量 Tool Calling。LLM 每一步输出 JSON action，系统只允许它调用 SkillRegistry 中的白名单工具，并记录每一步 trace。为了解决模型 JSON 不稳定，我加入了 repair prompt 和 deterministic fallback。

## 6.4 LangChain Agent Loop：ReAct 工具循环

适合：展示标准 LangChain Agent 能力。

调用链：

```text
load_memory_context
  -> profile_csv
  -> route_task
  -> langchain_agent_loop
      -> LangChainCSVAgent
          -> build_context(state)
          -> build_langchain_tools(registry, context)
          -> build ReAct prompt
          -> AgentExecutor 或 fallback executor
          -> Thought
          -> Action
          -> Observation
          -> repeat
          -> Final Answer
  -> finalize
  -> END
```

文件：

```text
src/agent/langchain_agent.py
src/agent/langchain_tools.py
```

核心设计：

```text
SkillRegistry -> DynamicSkillTool -> LangChain BaseTool
```

每个 Tool 调用时会自动补齐：

```text
run_id, csv_path, query, profile, charts, insights, markdown
```

ReAct 格式：

```text
Thought: 我需要先了解数据结构
Action: profile_csv
Action Input: {...}
Observation: {...}
Thought: 接下来推荐图表
Action: suggest_chart
Action Input: {...}
Observation: {...}
Final Answer: 中文总结，包含图表和报告路径
```

面试讲法：

> LangChain Agent Loop 是项目里最像标准智能体的模式。我把自定义 SkillRegistry 包装成 LangChain Tools，让模型通过 ReAct 的 Thought、Action、Observation 循环自主完成 CSV 分析、画图、生成洞察和导出报告。

## 7. 四种模式对比

| 模式 | 输出 | 稳定性 | 自主性 | 展示重点 |
|---|---|---:|---:|---|
| Quick Chart | 单图 + 解释 | 高 | 低 | 基础分析能力 |
| Full Report | 多图 + 报告 + PDF/HTML | 很高 | 中 | 完整产品闭环 |
| Planner Loop | JSON 工具规划 + Trace | 中高 | 中高 | Tool Calling 和 fallback |
| LangChain Agent Loop | ReAct Agent + 报告 | 中 | 高 | LangChain Agent 能力 |

推荐演示顺序：

```text
Full Report -> Planner Loop -> LangChain Agent Loop -> Quick Chart
```

## 8. Skill Registry 机制

文件：`src/skills/registry.py`

为什么需要：

```text
不能让 LLM 直接生成并执行 Python 代码，否则存在安全风险和不可控问题。
```

SkillRegistry 提供：

```text
register(skill)       注册工具
get(name)             查询工具
list_skills()         列出工具 schema
describe_skills()     给 LLM 的工具说明
call(name, **kwargs)  统一调用工具
get_call_log()        查看调用日志
```

异常处理：

```text
未知工具 -> structured error
工具异常 -> fallback_run
调用日志 -> 记录 skill、args、success、duration
```

## 10. MemoryStore 设计

存储文件：

```text
memory/task_history.jsonl
memory/user_profile.json
memory/chart_preference.json
memory/.cursor
```

记录内容：

```text
run_id, mode, csv_name, query, rows, columns,
chart_count, chart_types, chart_paths, report_path, success, error
```

作用：

1. 在任务开始前构造 `memory_context`；
2. 记录最近任务；
3. 统计图表偏好；
4. 为后续 Agent 推荐提供上下文。

面试讲法：

> MemoryStore 是一个轻量长期记忆模块。我没有一开始就引入向量数据库，而是先用 JSONL/JSON 保存历史任务和图表偏好，简单、可解释、易调试。后续可以升级为向量记忆。

## 11. LLMClient 设计

文件：`src/llm/client.py`

支持后端：

```text
DeepSeek -> OpenAI -> Ollama
```

核心方法：

```text
chat()                  普通文本生成
chat_json()             JSON 输出 + 重试
chat_json_with_trace()  JSON 输出 + 错误阶段追踪
as_langchain_chat_model() 转成 LangChain ChatModel
```

错误阶段：

```text
llm_call      LLM 调用失败
llm_parse     输出不是合法 JSON
validation    JSON 合法但 schema 不匹配
ok            成功
```

## 12. 报告导出和 fallback

报告流程：

```text
draft_markdown_report
  -> 生成 Markdown
export_pdf
  -> 保存 Markdown
  -> 渲染 HTML
  -> 尝试 WeasyPrint 导出 PDF
  -> PDF 失败则保留 HTML fallback
```

为什么需要 fallback：

```text
Windows 上 WeasyPrint 可能缺 GTK/Pango/Cairo 等系统依赖，所以不能让 PDF 失败导致整个工作流中断。
```

## 15. 面试高频问题

### Q1：为什么做这个项目？

答：我想做一个有真实任务闭环的 AI Agent 应用。CSV 分析是常见需求，传统流程需要手动看字段、选图表、写结论和整理报告，所以我用 LLM Agent 自动完成这些步骤。

### Q2：为什么不用 LLM 直接写 Python？

答：直接执行 LLM 生成的 Python 有安全和稳定性风险。我把能力封装成白名单 Skill，LLM 只能选择工具和参数，真正执行由受控代码完成。

### Q3：LangGraph 和 LangChain 分别做什么？

答：LangGraph 负责稳定工作流，比如 Full Report；LangChain 负责 Agent Loop，把 Skill 包装成 Tools，让模型通过 ReAct 自主调用工具。

### Q4：四种模式怎么区别？

答：Quick Chart 是单图模式；Full Report 是完整报告模式；Planner Loop 是 JSON 工具规划；LangChain Agent Loop 是标准 ReAct 工具循环。

### Q5：Planner Loop 怎么处理非法 JSON？

答：先记录 `llm_parse` trace，再追加 repair prompt。如果连续失败两次且已有中间产物，就 deterministic fallback，自动补 `generate_insight`、`draft_markdown_report`、`export_pdf`。

### Q6：MemoryStore 有什么用？

答：保存历史任务、图表类型、报告路径和用户偏好，后续分析时注入 memory context，让 Agent 能参考历史偏好。

### Q7：PDF 失败怎么办？

答：PDF 依赖 WeasyPrint 和系统库。项目会先保存 Markdown 和 HTML，再尝试 PDF。如果 PDF 失败，保留 HTML fallback，不中断流程。

### Q8：项目难点是什么？

答：主要是把 LLM 的不确定输出变成可控工具调用；处理 JSON 解析失败、工具失败、max_steps 用完等稳定性问题；以及把图表、洞察、报告导出串成完整闭环。

### Q9：怎么防止 LLM 编造数据？

答：统计值和图表都来自 pandas 对 CSV 的真实计算；LLM 只基于 profile、charts 和 insights 写结论；Full Report 还有 safety_check 做基本检查。

### Q10：后续怎么优化？

答：升级为原生 tool calling 或 structured output；限制 Agent 图表数量并去重；加入向量记忆；增加多 CSV 对比分析；部署在线 demo。
